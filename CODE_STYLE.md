# VectorKit 代码规范

## 1. 代码风格

### 1.1 Python 代码风格
- 遵循 [PEP 8](https://www.python.org/dev/peps/pep-0008/) 代码风格指南
- 使用制表符进行缩进
- 函数和类之间使用两个空行分隔
- 类内部方法之间使用一个空行分隔
- 运算符两侧使用空格，如 `a = b + c` 而非 `a=b+c`
- 逗号后使用空格，如 `def func(a, b, c):` 而非 `def func(a,b,c):`

### 1.2 导入规范
- 导入语句按以下顺序排列：
  1. 标准库导入
  2. 第三方库导入
  3. 本地模块导入
- 每组导入之间使用空行分隔
- 使用相对导入时，确保路径清晰
- 避免使用 `from module import *` 形式的导入

## 2. 命名约定

### 2.1 变量和函数
- 变量名使用小写字母，单词之间用下划线分隔，如 `user_name`
- 函数名使用小写字母，单词之间用下划线分隔，如 `calculate_average()`
- 常量使用全大写字母，单词之间用下划线分隔，如 `MAX_ITERATIONS`

### 2.2 类和模块
- 类名使用双下划线命名法，如 `__DataProcessor`
- 模块名使用小写字母，单词之间用下划线分隔，如 `signal_processor.py`
- 包名使用小写字母，单词之间不使用分隔符，如 `vectorkit`


### 2.3 私有成员
- 私有变量和方法使用单下划线前缀，如 `_private_method()`
- 强私有成员使用双下划线前缀，如 `__strong_private_method()`

## 3. 代码结构

### 3.1 文件结构
- 功能相关的代码应组织到同一个模块中
- 大型功能应拆分为多个模块，放置在相应的包中

### 3.2 函数和方法
- 函数应只做一件事情，并做好这件事情
- 函数应包含清晰的文档字符串，说明功能、参数和返回值

### 3.3 类设计
- 类应具有明确的职责，遵循单一职责原则
- 类的方法应围绕类的职责展开
- 避免创建过大的类，单个类代码行数不超过 200 行

## 4. 文档规范

### 4.1 文档字符串
- 所有模块、类、函数都应包含文档字符串
- 文档字符串应使用三引号 `"""` 包裹
- 模块文档字符串应说明模块的功能和用途
- 类文档字符串应说明类的职责和主要方法
- 函数文档字符串应说明函数的功能、参数、返回值和可能的异常

### 4.2 注释
- 代码中应添加适当的注释，解释复杂的逻辑
- 注释应使用中文，清晰明了
- 避免添加不必要的注释，如 `i = i + 1  # 加 1`
- 注释应与代码保持同步，避免注释与代码不一致

## 5. 测试规范

### 5.1 测试文件
- 测试文件应与被测试模块放在同一目录下，命名为 `test_模块名.py`
- 测试类应继承自 `unittest.TestCase`，使用双下划线命名法，如 `__TestClassName`
- 测试方法应以 `test_` 开头

### 5.2 测试覆盖
- 单元测试应覆盖主要功能和边缘情况
- 确保测试代码能够捕获常见的错误和异常
- 测试代码应与生产代码保持同步

## 6. 提交规范

### 6.1 提交信息
- 提交信息应清晰明了，包含以下内容：
  - 提交类型：如 `feat`（新功能）、`fix`（修复 bug）、`docs`（文档更新）、`style`（代码风格）、`refactor`（代码重构）、`test`（测试）、`chore`（构建/依赖）
  - 提交描述：简洁描述本次提交的内容
  - 详细说明：如有需要，可在描述后添加详细说明

- 提交信息示例：
  ```
  feat: 添加信号处理模块
  
  新增了信号处理模块，包含傅里叶变换、滤波等功能
  ```

### 6.2 分支管理
- 主分支：`master`，用于发布稳定版本
- 开发分支：`develop`，用于集成新功能
- 特性分支：`feat/功能名称`，用于开发特定功能
- 修复分支：`fix/问题描述`，用于修复特定问题

## 7. 代码审查

### 7.1 审查流程
- 代码提交前应进行自我审查
- 提交 Pull Request 后，至少需要一位团队成员进行审查
- 审查通过后，代码才能合并到主分支

### 7.2 审查重点
- 代码风格是否符合规范
- 代码逻辑是否正确
- 代码是否存在潜在的性能问题
- 代码是否存在安全隐患
- 文档是否完整

## 8. 性能优化

### 8.1 优化原则
- 优先保证代码的正确性和可读性
- 只在性能瓶颈处进行优化
- 优化前应进行性能分析，确定瓶颈所在
- 优化后应进行测试，确保优化效果

### 8.2 常见优化技巧
- 使用适当的数据结构，如列表、字典、集合等
- 避免不必要的计算和重复操作
- 使用生成器和迭代器减少内存使用
- 合理使用缓存机制

## 9. 安全规范

### 9.1 安全原则
- 避免使用不安全的函数和方法
- 对输入数据进行验证和 sanitization
- 避免硬编码敏感信息
- 遵循最小权限原则

### 9.2 常见安全问题
- SQL 注入
- 跨站脚本攻击（XSS）
- 跨站请求伪造（CSRF）
- 敏感信息泄露

## 10. 工具链

### 10.1 开发工具
- 代码编辑器：推荐使用 VS Code 或 PyCharm
- 版本控制：Git
- 包管理：pip

### 10.2 代码质量工具
- 代码风格检查：flake8
- 代码格式化：black
- 静态类型检查：mypy
- 测试框架：unittest 或 pytest

## 11. 附则

- 本规范将根据项目发展和实际情况进行调整
- 所有贡献者应遵守本规范
- 如有疑问，可在项目讨论区提出
